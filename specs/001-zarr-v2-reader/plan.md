# Implementation Plan: Zarr v2 Reader

**Branch**: `001-zarr-v2-reader` | **Date**: 2026-04-03 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/001-zarr-v2-reader/spec.md`

## Summary

Build a read-only Node.js library for Zarr v2 arrays with three storage
backends (FileSystem, HTTP, S3). Core pipeline: parse metadata -> resolve
chunk keys -> fetch via Store -> decode via Codec -> assemble TypedArray.
Extensible via Store and Codec interfaces.

## Technical Context

**Language/Version**: TypeScript 5.x with `strict: true`, targeting ES2022
**Primary Dependencies**: `node:zlib` (gzip), `node:fs/promises` (filesystem), native `fetch` (HTTP), `@aws-sdk/client-s3` (S3, peer dependency)
**Storage**: Zarr v2 stores on filesystem, HTTP servers, and S3 buckets (read-only)
**Testing**: vitest with `@vitest/coverage-v8`; Python zarr-generated fixtures
**Target Platform**: Node.js >= 22 (LTS), Linux/macOS/Windows servers
**Project Type**: Library (npm package)
**Performance Goals**: 100MB chunked array read from filesystem in < 2 seconds; slice reads of 1GB array with < 100MB library overhead
**Constraints**: ESM-only, minimal runtime dependencies, 10 concurrent chunk fetches default, retry 3x with exponential backoff for HTTP/S3
**Scale/Scope**: N-dimensional arrays up to multi-GB; single-consumer library (not multi-tenant service)

```text
src/
├── index.ts                  # Public API surface (re-exports)
├── store/
│   ├── store.ts              # Store interface definition
│   ├── filesystem.ts         # FileSystemStore
│   ├── http.ts               # HTTPStore
│   └── s3.ts                 # S3Store
├── metadata/
│   ├── v2.ts                 # Zarr v2 metadata types + parser
│   └── types.ts              # Shared metadata types
├── codec/
│   ├── codec.ts              # Codec interface + CodecRegistry
│   ├── raw.ts                # Raw (no-op) codec
│   └── gzip.ts               # Gzip codec (node:zlib)
├── chunk/
│   ├── loader.ts             # Chunk fetch orchestration + concurrency
│   └── indexing.ts           # Slice -> chunk index mapping
├── array.ts                  # ZarrArray class
├── group.ts                  # ZarrGroup class
├── dtype.ts                  # Dtype string -> TypedArray mapping + endianness
└── errors.ts                 # Custom error classes

tests/
├── fixtures/                 # Real Zarr v2 dirs (generated by Python)
│   ├── generate.py           # Fixture generation script
│   ├── simple_1d/            # 1D float32 array, no compression
│   ├── chunked_2d/           # 2D int32 array, multiple chunks
│   ├── compressed_gzip/      # Gzip-compressed array
│   ├── nested_groups/        # Group hierarchy with attrs
│   ├── big_endian/           # Big-endian float64 array
│   └── f_order/              # Fortran-order array
├── contract/
│   └── store.contract.ts     # Shared Store interface tests
├── unit/
│   ├── metadata.test.ts      # Metadata parsing
│   ├── dtype.test.ts         # Dtype mapping + endianness
│   ├── codec.test.ts         # Codec decode
│   └── indexing.test.ts      # Slice -> chunk indices
└── integration/
    ├── filesystem.test.ts    # FileSystemStore + real fixtures
    ├── http.test.ts          # HTTPStore + local test server
    ├── s3.test.ts            # S3Store + MinIO container
    └── array.test.ts         # Full pipeline: open -> read -> verify
```

**Structure Decision**: Single project layout. Library with `src/` for source and `tests/` for all test levels. Domain-driven subdirectories within `src/` (store, metadata, codec, chunk) reflecting the core abstractions.
