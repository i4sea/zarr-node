# zarr-node Constitution

## Core Principles

### I. Read-Only by Design

Every public API surface assumes read-only access to Zarr stores. Store
implementations MUST expose only three operations: `get(key)`, `has(key)`,
and `list(prefix)`. No write paths, no partial updates, no lock management.

This constraint simplifies the entire architecture: fewer failure modes,
no concurrency conflicts, no transactional guarantees needed. Write support
MAY be introduced in a future major version but MUST NOT leak into v1
interfaces, types, or abstractions.

### II. TypeScript-First

All source code MUST be authored in TypeScript with `strict: true` enabled.
Public API signatures MUST NOT contain `any`. Generic type parameters MUST
flow from array dtype metadata through to the returned `TypedArray` subtype
(e.g., a `<f4` dtype resolves to `Float32Array`).

Types are the primary documentation. Every exported function, class, and
interface MUST have accurate type signatures that make incorrect usage a
compile-time error rather than a runtime surprise.

### III. Test-Driven Development (NON-NEGOTIABLE)

TDD is mandatory for all production code: tests written first, confirmed
failing (red), then implementation (green), then refactor. This cycle MUST
NOT be skipped or reordered.

- Every Store implementation MUST pass a shared contract test suite.
- Every Codec MUST have decode tests using known input/output byte pairs.
- Integration tests MUST use real Zarr fixture files generated by the
  reference Python `zarr` library to ensure cross-implementation correctness.
- Coverage regressions MUST block CI.

### IV. Extensible Plugin Architecture

Stores and codecs are registered through well-defined TypeScript interfaces,
not hardcoded. The core library ships built-in implementations but MUST NOT
assume they are the only ones.

- Users MUST be able to implement the `Store` interface for custom backends
  (e.g., GCS, Azure Blob, database) and use them with the library.
- Users MUST be able to register custom `Codec` implementations via a
  public codec registry.
- Plugin interfaces MUST be stable across minor versions.

### V. Server-First Performance

This library is designed for Node.js server workloads: large arrays,
concurrent chunk reads, and memory-conscious operation.

- Use Node.js-native primitives (`Buffer`, `node:fs/promises`, `node:zlib`,
  streams) rather than browser polyfills or abstractions.
- Multi-chunk reads MUST support configurable concurrency limits to prevent
  resource exhaustion.
- Performance benchmarks for critical paths (chunk decode, multi-chunk read)
  MUST be tracked and regressions MUST be detected in CI.

### VI. Semantic Versioning and API Stability

The project follows strict semantic versioning (semver). The public API is
defined by exported types and function signatures.

- Breaking changes MUST require a major version bump.
- Deprecations MUST be warned for at least one minor release before removal.
- Every PR that modifies public API MUST update the CHANGELOG.
- Pre-1.0 releases (0.x.y) MAY break API in minor versions but MUST
  document changes clearly.

### VII. Simplicity (YAGNI)

Start with the smallest useful API. Every addition MUST be justified by a
concrete user scenario, not speculation.

- Prefer composition over inheritance.
- Prefer functions over classes where the abstraction does not need state.
- Three similar lines of code are better than a premature abstraction.
- Do not add error handling for scenarios that cannot happen.
- Do not design for hypothetical future requirements.

## Technical Constraints

- **Runtime**: Node.js >= 22 (LTS). No browser support in v1.
- **Module System**: ESM-only (`"type": "module"` in package.json).
- **Dependencies**: Minimize runtime dependencies. Use Node.js built-ins
  wherever possible (`node:zlib`, `node:fs/promises`, native `fetch`).
  Native codec bindings (e.g., blosc) are acceptable for performance.
  Cloud SDK dependencies (e.g., `@aws-sdk/client-s3`) MUST be peer
  dependencies, not direct dependencies.
- **Build Target**: ES2022. Output includes declaration files and
  declaration maps for downstream TypeScript consumers.
- **License**: MIT.

## Development Workflow

- **Branching**: Sequential branch numbering (e.g., `001-core-store`,
  `002-filesystem-backend`). Feature branches merge to `main`.
- **Pull Requests**: All changes require PR with review. Direct pushes
  to `main` are prohibited.
- **CI Pipeline**: Every PR MUST pass: TypeScript type-check, ESLint,
  Prettier format check, and full test suite on Node.js 22 and 24.
- **Changelog**: Managed via changesets. Every user-facing change MUST
  include a changeset entry.
- **Commit Style**: Conventional commits (e.g., `feat:`, `fix:`, `test:`,
  `docs:`, `chore:`).

## Governance

This constitution supersedes all other practices, conventions, and ad-hoc
decisions within the project.

- **Compliance**: All PRs and code reviews MUST verify compliance with
  these principles. Non-compliance MUST be flagged and resolved before merge.
- **Amendments**: Changes to this constitution require:
  1. A written proposal documenting the change and its rationale.
  2. Review and approval.
  3. A migration plan for any existing code affected by the change.
  4. Version bump of this constitution per semver rules (MAJOR for principle
     removal/redefinition, MINOR for additions, PATCH for clarifications).
- **Complexity Justification**: Any deviation from the Simplicity principle
  (VII) MUST be documented in the plan's Complexity Tracking section with
  rationale and rejected alternatives.
